Content Broadcasting System - Enhanced Architecture Notes

Authentication & RBAC Flow
- JWT-based authentication with 24-hour expiration
- Role-based access control: Principal and Teacher roles
- Middleware protects routes based on user roles
- Password hashing using bcryptjs (10 salt rounds)
- Token verification on each protected request
- Centralized error handling for authentication failures

Subject-based System Design
- Content organized by subjects (Maths, Science, etc.)
- Each subject has independent rotation logic
- Subjects rotate every 30 minutes in public API
- Content within subjects rotates every 5 minutes
- Teacher-specific content isolation
- Analytics tracking for subject performance

Enhanced Upload Handling Approach
- Multer middleware with dual storage support
- Local storage in uploads directory (fallback)
- AWS S3 cloud storage (when configured)
- File validation: JPG, PNG, GIF only
- Maximum file size: 10MB (configurable)
- Unique filename generation to prevent conflicts
- Automatic storage switching based on configuration
- File metadata stored in database with analytics tracking

Approval Workflow Design
- Teachers upload content with "uploaded" status
- Principal can view all pending content
- Principal can approve or reject content
- Rejection requires reason
- Approved content becomes available for broadcasting
- Status flow: uploaded -> pending -> approved/rejected
- Cache invalidation on approval/rejection
- Analytics tracking for content views

Advanced Scheduling/Rotation Logic (CRITICAL)
- Time-based content activation using start_time and end_time
- Subject rotation: 30-minute intervals
- Content rotation within subject: 5-minute intervals
- Current time determines active content
- Redis caching for performance optimization
- Graceful handling of no content scenarios
- Teacher-specific scheduling isolation
- Real-time view tracking for analytics

Enhanced Database Design
- MySQL for relational data integrity
- Users table with role constraint
- Content table with status enum and scheduling fields
- Content_analytics table for view tracking
- Foreign key relationships with cascade delete
- Database indexing for performance optimization
- Timestamp fields for audit trail
- Content slots and schedule tables for future expansion

Configuration Management
### Environment Variables
- Database credentials and connection settings
- JWT secret for authentication
- File upload directory and size limits
- Redis configuration for caching (optional)
- AWS S3 credentials for cloud storage (optional)

### Optional Features Configuration

#### Redis Caching (Optional)
- **Disabled by default**: `REDIS_ENABLED=false`
- **Enable**: Set `REDIS_ENABLED=true` and start Redis service
- **Behavior**: Caches public API responses for 60 seconds
- **Fallback**: System continues without caching if Redis unavailable
- **Performance**: Improves response times for public endpoints

#### AWS S3 Storage (Optional)
- **Disabled by default**: `S3_ENABLED=false`
- **Enable**: Set `S3_ENABLED=true` and provide AWS credentials
- **Behavior**: Stores uploaded files in cloud storage
- **Fallback**: Automatic local storage if S3 fails
- **File Access**: Provides public URLs for cloud-stored files

#### Default Behavior
When both features are disabled:
- Files stored locally in `uploads/` directory
- Direct database calls (no caching)
- All core features fully functional
- No external dependencies required
